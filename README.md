# ⚡ AgentOS — AI Organization Operating System

AgentOS is a dynamic AI Organization Operating System that allows anyone to describe a company, task, or goal in plain language, and automatically generates a hierarchy of specialized AI agents that collaborate in real-time to execute that goal.

Unlike developer frameworks that require complex setups, or expensive enterprise agent environments, AgentOS fills the consumer gap: a zero-code, lightweight platform where users can deploy a collaborative AI organization in under 5 minutes.

---

## 🚀 Key Features

*   **Dynamic Role Generation**: The system infers necessary agent roles (e.g., CEO, CTO, CFO, Legal, Product) directly from your plain-text project description using LLM logic.
*   **Parallel DAG Execution**: Independent strategist agents (Layer 0) execute in parallel first. Dependent execution agents (Layer 1) start automatically once strategist outputs are resolved.
*   **Inter-Agent Communication Message Bus**: Agents automatically exchange inquiries (`QUESTION_TO` / `ANSWER_TO`) to align on project requirements (e.g., CTO answering PM's questions on security protocols).
*   **Real-time WebSocket Streaming**: Live outputs are streamed character-by-character from each agent, allowing the user to watch the organization collaborate in real-time.
*   **Interactive Visualizer**: A dynamic SVG-connected organizational chart rendering status indicators (Pending, Running, Waiting, Done, Error) and selected AI models.
*   **Session Persistence**: All execution history, agent configurations, inter-agent messages, and final consolidated reports are persisted in SQLite, allowing users to return and load previous sessions easily.

## 🧠 Codebase Architecture & Deep Dive

This section completely explains the internals of AgentOS to help developers understand the architecture and contribute.

### 1. Database Schema (SQLite)
AgentOS uses SQLite for persistent storage, managed in `backend/agentos.db`. The schema consists of four core tables:
- **`sessions`**: Represents a single project execution. Stores the user's initial prompt (`description`), status (draft/running/completed), and timestamps.
- **`agents`**: The spawned AI entities. Each agent belongs to a session, has a specific `role` (e.g., CEO, CTO), assigned LLM `model`, `system_prompt`, `task`, and belongs to a specific dependency `layer` (0 for strategists, 1 for executors).
- **`messages`**: Acts as the inter-agent communication bus. Records direct messages (`QUESTION_TO` or `ANSWER_TO`) between `from_agent` and `to_agent` to simulate realistic team collaboration.
- **`results`**: Caches the final consolidated outputs, syntheses, and key metrics extracted from the agent collaboration.

### 2. Backend Engine (`backend/main.py`)
The backend is built with **FastAPI** leveraging Python's `asyncio` for non-blocking concurrent operations.

- **Dynamic Agent Generator**: When a session is created (`POST /api/sessions`), the engine calls the Gemini API (or falls back to a mock generator) with the user's description. The LLM infers what AI roles are needed (Layer 0 strategists vs. Layer 1 executors) and persists them in the `agents` table.
- **DAG Execution Engine**: When a session is run (`POST /api/sessions/{session_id}/run`), `execute_dag` groups agents by their `layer`. Layer 0 agents run concurrently in the background. Once they finish, Layer 1 agents launch, consuming the context generated by Layer 0. This simulates a real corporate hierarchy where executives plan and managers execute.
- **WebSocket Manager**: Provides the real-time token streaming (`/ws/sessions/{id}`). As the background tasks simulate LLM output generation (or stream real LLM tokens), they use `ConnectionManager.broadcast()` to push live JSON updates to connected browser clients.

### 3. Frontend Architecture (`frontend/`)
The frontend is built using Vanilla HTML5, CSS3, and ES6 Modules—no heavy frameworks—for maximum performance.

- **`index.html`**: The landing page and session creator. Users describe their goal here.
- **`session.html`**: The core execution dashboard.
  - **`ws.js`**: Connects to the FastAPI WebSocket. Listens for `AGENT_STATUS` updates (to update the UI) and `TOKEN_CHUNK` broadcasts (to print live text in the agent panels).
  - **`orgChart.js`**: A dynamic visualizer that builds SVG lines (`<path d="..." />`) connecting Layer 0 to Layer 1 agents, visually representing the Directed Acyclic Graph (DAG) dependencies.
  - **`agentPanel.js`**: Manages the slide-out panel allowing users to inspect an agent's individual token stream, system prompt, and manually override the assigned LLM model via REST API (`PATCH /api/agents/{id}/model`).
- **`results.html`**: Fetches the synthesized final blueprint (`GET /api/sessions/{id}/results`) and renders it with charts and downloadable artifacts (`GET /api/sessions/{id}/export`).

## 🏁 Quick Start Guide

### 1. Prerequisites
Ensure you have the following installed on your machine:
*   [Python 3.11 or higher](https://www.python.org/downloads/)
*   An API Key from [Google AI Studio](https://aistudio.google.com/) (Free tier available)

### 2. Clone the Repository
```bash
git clone https://github.com/your-username/AgentOS.git
cd AgentOS
```

### 3. Backend Setup
1.  Navigate to the `backend` directory:
    ```bash
    cd backend
    ```
2.  Create a virtual environment (optional but recommended):
    ```bash
    python -m venv venv
    # On Windows:
    .\venv\Scripts\activate
    # On macOS/Linux:
    source venv/bin/activate
    ```
3.  Install dependencies:
    ```bash
    pip install -r requirements.txt
    ```
4.  Configure Environment Variables:
    Create a `.env` file in the root of the project (or inside `backend/` directory) and populate it (see `.env.example`):
    ```env
    GEMINI_API_KEY=your_gemini_api_key_here
    ```
5.  Start the backend server:
    ```bash
    python main.py
    # OR:
    python -m uvicorn main:app --host localhost --port 8000 --reload
    ```
    The FastAPI server will start at `http://localhost:8000`.

### 4. Frontend Setup
You can serve the static frontend folder using any web server. For example:
*   Using Python's built-in server:
    ```bash
    # From the project root:
    python -m http.server 3000 --directory frontend
    ```
*   Or using Node's `http-server` utility:
    ```bash
    npx http-server frontend -p 3000
    ```
Open `http://localhost:3000` in your browser.

## 📡 API Endpoints Reference

| Method | Endpoint | Description |
| :--- | :--- | :--- |
| `POST` | `/api/sessions` | Create a new session from user goals and dynamically spawn agents. |
| `GET` | `/api/sessions` | Fetch all historical sessions. |
| `GET` | `/api/sessions/{id}` | Load details of a specific session (including its agents list). |
| `POST` | `/api/sessions/{id}/run` | Start the DAG execution engine for a session. |
| `PATCH` | `/api/agents/{id}/model` | Override the assigned LLM model for a specific agent. |
| `GET` | `/api/sessions/{id}/results` | Fetch the final generated business blueprint/report. |
| `GET` | `/api/sessions/{id}/export` | Export the report as downloadable `.json` or `.md`. |
| `DELETE`| `/api/sessions/{id}` | Remove a session and its associated data from DB. |
| `WS` | `/ws/sessions/{id}` | WebSocket route for live agent status and tokens broadcasts. |
