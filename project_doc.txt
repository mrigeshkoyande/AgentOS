Agent Lifecycle & OrchestrationThe Global OrchestratorThe Global Orchestrator is the master AI brain of AgentOS. It is the first AI called when a user submits a description. Its responsibilities:Parse user input to understand the domain, goal, and contextDetermine what roles/agents are needed (e.g., CEO, CTO, Marketing, Legal)Define each agent's personality, expertise level, constraints, and model preferenceBuild the task dependency graph (DAG) — who depends on whomAssign each agent a specific, focused task aligned with the overall goalKey Insight: The user never manually creates or configures agents. The Global Orchestrator handles everything — agent creation, fine-tuning, task assignment, and dependency mapping.Agent Lifecycle FlowSTEP 1: USER SUBMITS DESCRIPTION  Input: "Build me an e-commerce startup plan"  └─▶ POST /api/sessions/createSTEP 2: GLOBAL ORCHESTRATOR RUNS  Prompt → Kimi K2 (best for reasoning/structure)  Output: { agents: [{ role: "CEO", model: "kimi-k2", task: "..." }, ...] }STEP 3: AGENT FACTORY CREATES AGENTS  For each agent definition:    → Build system prompt (role + personality + constraints)    → Assign model (AI-chosen, user can override)    → Register in DB with session_idSTEP 4: DAG EXECUTION ENGINE STARTS  Layer 1 → All independent agents execute in parallel  Layer 2 → Dependent agents start after Layer 1 completes  Between layers → Agents may send QUESTION_TO: [AgentX] messagesSTEP 5: RESULT AGGREGATION  → Collect all agent outputs  → Final synthesis by Global Orchestrator  → Export: JSON + Markdown + Text (PDF optional)Agent Object Structure{  agent_id:      string,       // unique UUID  session_id:    string,       // parent session  role:          string,       // e.g. 'CTO', 'Marketing Lead'  display_name:  string,       // e.g. 'Alex Chen - CTO'  model:         string,       // assigned AI model  model_override:string|null,  // user-set override  system_prompt: string,       // full tuned prompt  task:          string,       // specific task assigned  dependencies:  string[],     // agent_ids this depends on  layer:         number,       // DAG layer (0, 1, 2...)  status:        enum,         // pending|running|waiting|done|error  output:        string|null,  // result when done  messages:      Message[],    // inter-agent communications  created_at:    datetime,  completed_at:  datetime|null}5. Inter-Agent Communication ProtocolAgents communicate through a structured message protocol. When an agent needs information from another agent, it embeds a question in its response using a standardized format. The Message Bus parses, routes, and delivers these messages.Message Format# Agent embeds questions in its output:QUESTION_TO:[CTO]What is the maximum API throughput we can support in the first month?END_QUESTION# Agent embeds answers:ANSWER_TO:[CEO]Re: API throughput — We can handle up to 10,000 requests/day with current budget.END_ANSWER# Message object structure:{  message_id:   string,  session_id:   string,  from_agent:   string,   // agent_id  to_agent:     string,   // agent_id    type:         'question' | 'answer' | 'info',  content:      string,  timestamp:    datetime,  resolved:     boolean}Message Bus ArchitectureMESSAGE BUS FLOW:  CEO Agent         Message Bus         CTO Agent     │                   │                    │     │── QUESTION_TO ──▶│                    │     │                   │── route ──────────▶│     │                   │                    │ (CTO processes)     │                   │◀── ANSWER_TO ──────│     │◀── deliver ────────│                    │     │                   │                    │     │ (CEO re-runs with answer injected)        │WebSocket broadcasts ALL messages → Frontend (visible to user)Communication RulesMax 3 question rounds per agent to prevent infinite loops60-second timeout per question — if no answer, agent proceeds without itCircular dependencies blocked — Agent A cannot wait on Agent B if B waits on AAll messages stored in DB — visible in session historyWebSocket broadcasts every message in real-time to frontendPriority: Higher-layer agents get priority in message queue7. Parallel Execution EngineThe Parallel Execution Engine runs agents in optimized layers. Agents within the same layer execute simultaneously using Python asyncio. A new layer starts only after the previous layer fully completes.DAG Layer ExampleINPUT: "Plan my SaaS startup"LAYER 0 — Independent Agents (all run in parallel):  ┌─────────┐  ┌─────────┐  ┌──────────┐  ┌─────────┐  │   CEO   │  │   CTO   │  │  CFO     │  │ Mktg    │  │ running │  │ running │  │ running  │  │ running │  └────┬────┘  └────┬────┘  └────┬─────┘  └────┬────┘       └──────────────┴────────────┴──────────────┘                           │ all done                 LAYER 1 — Dependent Agents (need Layer 0 outputs):  ┌──────────┐  ┌──────────┐  ┌──────────────────┐  │ Dev Lead │  │  Legal   │  │ Product Manager  │  │ (uses    │  │ (uses    │  │ (uses CEO+CTO    │  │  CTO)    │  │  CFO)    │  │  outputs)        │  └────┬─────┘  └────┬─────┘  └────────┬─────────┘       └──────────────┴─────────────────┘                           │                         LAYER 2 — Final Synthesis:  ┌──────────────────────────────────────┐  │  CEO — Final Consolidated Report      │  └──────────────────────────────────────┘Core Execution Code Patternasync def execute_dag(session_id: str, agents: list[Agent]):    layers = build_layers(agents)  # Group agents by dependency depth        for layer_num, layer_agents in enumerate(layers):        # Broadcast layer start to WebSocket        await ws_broadcast(session_id, {            "event": "layer_start",            "layer": layer_num,            "agents": [a.role for a in layer_agents]        })                # Run all agents in this layer simultaneously        tasks = [execute_agent(agent, session_id) for agent in layer_agents]        results = await asyncio.gather(*tasks, return_exceptions=True)                # Process inter-agent messages from this layer        await process_messages(session_id, layer_agents)            # Aggregate all outputs into final result    return await aggregate_results(session_id, agents)8.
